# 记忆工具与混合检索

02 篇说了「冷数据靠检索」。这一篇拆检索侧的实现：两个工具、混合检索算法、分块与去重。

这部分细节大量来自社区把 OpenClaw 记忆检索抽取并开源的 **memsearch** 项目分析，属于对实现的逆向描述，标 **【社区拆解】**；官方文档确认的标 **【事实】**。

## 1. 两个工具

**【事实】** 记忆检索由当前激活的 memory 插件提供（默认 `memory-core`），暴露两个工具：

| 工具 | 作用 |
|---|---|
| `memory_search` | 语义检索：即使措辞和原文不同也能命中（hybrid 向量 + 关键词）|
| `memory_get` | 精确读取：按文件路径 / 行范围读，缺失文件优雅返回空文本 |

二者覆盖 `MEMORY.md` + 所有 daily logs。检索协议（写在 `AGENTS.md` 里）要求：

```text
Before doing non-trivial work:
  memory_search for the project / topic / user preference
```

即「动手前先检索」。`memory_get` 的「缺失即返回空」是刻意的容错——让模型可以乐观地去读一个可能不存在的文件而不必先判存在性，减少一次工具往返。

## 2. 混合检索：向量 70% + 关键词 30%

**【事实 / 社区拆解】** 当配置了 embedding provider 时，`memory_search` 用 **hybrid search**：向量相似度（语义）+ 关键词匹配（精确词，如 ID、代码符号）。权重：

```json
{ "hybrid": { "vectorWeight": 0.7, "textWeight": 0.3 } }
```

```text
向量检索 (70%)  解决「语义相近、措辞不同」
  query "Redis cache config" 命中 "Redis L1 cache with 5min TTL"

BM25 关键词 (30%)  解决「必须精确命中」
  版本号、错误码、函数名——近似会出错的地方
```

**为什么是混合而不是纯向量？** 纯 embedding 在「语义模糊查询」上强，但对「精确 token」很差：你查错误码 `E1234` 或函数名 `buildMemorySection`，向量空间里它们和一堆语义相近但无关的东西混在一起。BM25 这类稀疏检索恰好擅长精确词命中。混合 = 两种失败模式互补。70/30 的偏向说明 OpenClaw 的记忆查询以「找语义相关的笔记」为主，精确匹配是兜底。

**【事实】** embedding provider 可配，默认 OpenAI，也支持 Gemini、Voyage、Mistral、本地、Ollama、Bedrock、GitHub Copilot 或 OpenAI 兼容端，通过 `agents.defaults.memorySearch.provider` 切换。

## 3. 分块：按标题语义切，不按字数硬切

**【社区拆解】** 索引时采用**语义分块（semantic chunking）**，沿 Markdown 标题边界切分，让相关内容留在同一块里：

```text
## Redis Caching
... 这一整段作为一个 chunk, 不会被按固定字数从中间劈开
```

**为什么不按固定 token 数硬切？** 固定窗口切分会把一个完整概念劈成两半，导致：检索时只命中半截、上下文不完整、向量表示被污染。按标题切让每个 chunk 是一个「自洽的语义单元」，这对检索质量的提升是结构性的——它利用了 Markdown 本身就是人写的、带语义结构的这一事实。

## 4. 去重与失效：SHA-256 + model_version

**【社区拆解】** 用 SHA-256 防止重复 embedding。chunk ID 编码为：

```text
chunk_id = hash(source_path : start_line : end_line : content_hash : model_version)
```

这个 ID 设计一举两得：
- **内容没变就不重算 embedding**（content_hash 相同 → 命中缓存），省钱省时
- **embedding 模型升级时自动识别陈旧块**（model_version 变了 → 旧 chunk_id 全部失效 → 触发重建）

这是把「索引一致性」做进了 ID 本身，而不是靠外部版本表维护，很干净。

## 5. 检索结果的呈现

**【社区拆解】** 返回 top-K（默认 3）chunk，初始截断到 200 字符，需要时再按需展开全文。

```text
memory_search(query)
  -> top 3 chunks
  -> 每块先给 200 字预览
  -> 模型判断相关后, 用 memory_get 拉全文
```

这是又一次「省 context」：搜索先给摘要，模型筛过再拉全文，避免一次把 K 个长块全灌进 context。和 02 篇「常驻最小化」是同一种预算意识，贯穿到了工具粒度。

## 6. 两条检索 track

**【社区拆解】** masterclass 提到两条后端：

| | Track A（内置）| Track B（QMD）|
|---|---|---|
| 后端 | 本地 embedding 模型 `embeddinggemma-300m-qat-Q8_0.gguf` + hybrid | Query Markdown Documents |
| 适用 | workspace 记忆文件，可加 `extraPaths` 挂外部 Markdown 而不换后端 | 搜 Obsidian vault、session transcript、大文档集 |
| 默认模式 | 向量+BM25 混合 | 默认 BM25 关键词 |

> 设计含义：内置 track 用一个量化过的小本地 embedding 模型（300m，Q8 量化），意味着**不配云端 embedding 也能用语义检索**——隐私和离线友好。Track B 则把「搜大文档库」和「搜记忆」解耦，避免把海量外部文档塞进记忆索引拖慢核心检索。

## 7. 一个已知的实现坑：检索上下文膨胀

**【事实】** GitHub issue #66060「active-memory: bloated prompt」记录了一个问题：完整的 agent system prompt 被带进了 memory 搜索上下文，导致 prompt 膨胀。

> 这条值得记住：**「记忆检索本身也要花 context」**。如果检索调用携带了过大的上下文（比如把整个 system prompt 也带上），检索这个本该省 token 的动作反而变贵。这说明冷热分层的省钱效果，依赖每个工具调用自身也保持精简——边界很容易在工程实现里漏掉。

## 8. 小结

```text
memory_search:
  hybrid(vector 0.7 + BM25 0.3)
  -> semantic chunking(按标题)
  -> SHA-256 去重 + model_version 失效
  -> top-K=3, 先 200 字预览
memory_get:
  按路径/行范围精确读, 缺失返回空
真相: 永远是文件; 索引只是可重建的加速层
```
