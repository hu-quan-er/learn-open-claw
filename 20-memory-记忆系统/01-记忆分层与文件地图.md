# 记忆分层与文件地图

OpenClaw 的记忆全部落在 **agent workspace**（默认 `~/.openclaw/workspace`）下，用纯 Markdown 文件承载。注意 workspace 和凭证目录是分开的：workspace 放记忆/人格文件，`~/.openclaw/` 下放 secrets，二者刻意隔离。

这一篇先把「有哪些文件、各存什么、为什么这么切」讲清楚。后面几篇讲它们怎么进出 context。

## 0. 为什么是 Markdown 文件，而不是数据库

**【事实】** 文档反复强调一句第一原理：

```text
The model only "remembers" what gets saved to disk — there is no hidden state.
"If it's not written to a file, it doesn't exist."
```

选 Markdown 文件而不是向量库/SQLite 作为「唯一真相」，带来三个直接好处（06 篇会展开理论依据）：
- **可审计**：人可以直接打开读、直接编辑
- **可版本化**：可以用 git 管理、diff、回滚
- **可调试**：可以 grep，把 AI 记忆当文档对待

向量索引是后来「叠」在文件之上的加速层，**索引不是真相，文件才是**。这点决定了整套系统的形态。

## 1. 两类文件：身份层 vs 记忆层

workspace 里的 Markdown 大致分两类用途，容易混淆，先分清：

### 1.1 身份/上下文层（bootstrap 每次重载）

**【事实】** 这些文件在每次会话启动时从磁盘加载，靠「重载」而非「留在对话历史里」来跨 compaction 存活：

| 文件 | 角色 |
|---|---|
| `SOUL.md` | 身份、语气、边界（人格自我图式）。注意它定义边界但**不强制安全策略** |
| `USER.md` | 用户上下文：用户是谁、在做什么项目、沟通偏好、技术环境 |
| `AGENTS.md` | 运行指令 / 工作流规则 / 工具约定 / **记忆协议本身** |
| `TOOLS.md` | 本地工具使用约定 |
| `IDENTITY.md` / `HEARTBEAT.md` / `BOOTSTRAP.md` | 其它 bootstrap 文件（身份、心跳、首启引导） |

它们不是「记忆」本体，但它们规定了 agent **如何使用记忆**——尤其 `AGENTS.md`，下面单独讲。

### 1.2 记忆层（本专题主角）

| 文件 | 层次 | 内容 |
|---|---|---|
| `MEMORY.md` | 长期 / 冷 | 「durable facts, preferences, decisions」——经过策展的稳定结论与摘要 |
| `memory/YYYY-MM-DD.md` | 工作 / 热 | 「detailed daily notes, observations, session summaries, raw context」——每天一个文件 |
| `memory/YYYY-MM-DD-<slug>.md` | 工作 / 热 | 带 slug 的当日变体，由 `/new`、`/reset` 的 session-memory hook 产生，和纯日期文件一起被识别 |
| `DREAMS.md` | 固化产物 | dreaming sweep 的人读摘要（Dream Diary），含历史回填条目 |
| `memory/.dreams/` | 机器状态 | dreaming 的内部状态（非人读） |
| `memory/dreaming/<phase>/YYYY-MM-DD.md` | 固化报告 | 可选的各阶段报告 |

**【事实】** 一个历史兼容细节：小写 `memory.md` 是 legacy 修复输入，不应和根目录的 `MEMORY.md` 同时刻意保留两份。

## 2. 长期 vs 工作：为什么要切成两层

这是整套设计里最核心的一刀。**【事实】** 文档明确的分工：

```text
MEMORY.md          = 紧凑、策展过的一层，放「standing decisions and summaries」
memory/*.md (daily) = 详细的工作层，放「raw context for later distillation」
```

并且明确了它们之间的流动方向：

```text
随着时间推移，agent 被期望:
  - 从 daily notes 里「蒸馏(distill)」出有用材料 -> 写进 MEMORY.md
  - 删除 MEMORY.md 里过期的长期条目
```

为什么不合成一个文件？因为这两层在「写入频率」和「注入频率」上是反向的：

```text
                  写入频率    注入到 prompt 的频率
daily notes        高(随手记)   低(只今天+昨天, 其余靠检索)
MEMORY.md          低(策展后)   高(每次会话都注入)
```

如果合并：要么把高频原始记录全注入，撑爆 context 预算；要么把策展结论也藏起来按需检索，丢掉「每次会话都该记得的硬约束」。切两层正是为了让**「常驻注入的那份」保持小而高信号**，把「体量大但低频用」的细节挪到检索层。这正是 02 篇要展开的冷热分层。

> **【推断】** `MEMORY.md` 之所以被建议「保持在 100 行以内 / 控制在 bootstrap 预算内」，本质就是它是常驻注入项，体积直接换算成每一轮对话的 token 成本。daily notes 没有这个约束，因为它们默认不进常驻 prompt。

## 3. `AGENTS.md` 里的记忆协议

**【事实】** 默认 `AGENTS.md` 把记忆的「读写时机」写成了显式协议，等于把记忆系统的使用契约投射给模型：

会话启动的读取顺序：
```text
Read `SOUL.md`, `USER.md`, and today+yesterday in `memory/`.
Read `MEMORY.md` when present.
```

写入纪律：
```text
- Before writing memory files, read them first;
  write only concrete updates, never empty placeholders.
- Capture: decisions, preferences, constraints, open loops.
- Avoid secrets unless explicitly requested.
- 日志写 memory/YYYY-MM-DD.md（目录不存在则创建）
- 长期事实/偏好/决定写 MEMORY.md
```

社区最佳实践里还有一条很关键：在 `AGENTS.md` 里加一句 **「search memory before acting」**——因为 `AGENTS.md` 本身会跨 compaction 存活，把「先查记忆」写进它，就等于让「检索习惯」也变成持久协议，而不是靠模型每次自觉。

> 设计含义：**记忆系统的「策略」不写死在代码里，而是写在用户可改的 `AGENTS.md` 里。** 这让用户可以按需收紧/放松记忆纪律（比如要求每次动作前必检索），而不需要改 OpenClaw 本体。这与 19 篇里「system prompt 把工具治理显式投射给模型」是同一种设计哲学。

## 4. 文件地图速查

```text
~/.openclaw/                         # 凭证/secrets（与记忆隔离）
~/.openclaw/workspace/
├── SOUL.md                          # 身份/语气/边界        ┐
├── USER.md                          # 用户上下文            │ bootstrap
├── AGENTS.md                        # 运行指令 + 记忆协议    │ 每次重载
├── TOOLS.md                         # 工具使用约定          ┘
├── MEMORY.md                        # 长期记忆(冷, 常驻注入)
├── DREAMS.md                        # dreaming 人读摘要
└── memory/
    ├── 2026-06-07.md                # 当日工作记忆(热)
    ├── 2026-06-06.md                # 昨日(启动时注入)
    ├── 2026-06-05.md                # 更早(仅检索可见)
    ├── 2026-06-07-handoff.md        # slug 变体(hook 产生)
    ├── .dreams/                     # dreaming 机器状态
    └── dreaming/<phase>/...         # dreaming 阶段报告
```
