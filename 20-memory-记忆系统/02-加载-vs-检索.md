# 加载 vs 检索：冷热分层与 context 预算

这一篇讲整套记忆系统在工程上最关键的折中：**哪些记忆每轮都塞进 prompt（常驻注入），哪些只在需要时才捞出来（按需检索）**。这背后是一个硬约束——context window 是有限且付费的。

## 1. 两种进入 context 的方式

**【事实】** OpenClaw 的记忆有两条完全不同的入场路径：

```text
(A) Bootstrap 注入 / 常驻
    会话启动时自动进入 model context:
      - MEMORY.md            (长期记忆全量, 超预算则截断)
      - memory/<today>.md    (今天的 daily)
      - memory/<yesterday>.md(昨天的 daily)

(B) 按需检索 / 不常驻
    运行中由模型主动调用工具捞取:
      - memory_search  语义+关键词混合检索
      - memory_get     按文件/行范围精确读取
    覆盖范围: 所有 daily notes（含历史）都被索引, 但不进 bootstrap
```

一句话概括：**「最近 + 最稳定」常驻，「久远 + 海量」按需。**

## 2. 为什么不全量注入：context 是稀缺资源

**【事实】** masterclass 把 LLM context window 描述成一个「所有组件抢空间」的容器（约 200K token）：system prompt、workspace 文件、对话历史、tool 调用与结果，全在里面竞争。

如果把全部历史 daily notes 都注入：
- token 成本随记忆累积**线性增长**，每一轮对话都要重付一次
- 真正相关的少量信息被海量无关历史稀释（“lost in the middle”式的检索退化）
- 撑满 context 会更早触发 compaction，反而加速记忆丢失

所以设计选择是：**把「连续性」需求（最近发生了什么、有哪些硬结论）用小体积常驻满足，把「偶尔需要翻旧账」的需求用检索满足。**

**【事实】** 文档对 daily notes 的定位说得很直白：

```text
daily 文件被 memory_search / memory_get 索引,
但不会在每一轮被注入到正常的 bootstrap prompt。
```

```text
keeps detailed logs in memory/YYYY-MM-DD.md so memory tools can
retrieve them on demand without injecting them into every prompt.
```

这就是「冷热分层」：热数据（今天/昨天 + 长期结论）常驻，冷数据（历史细节）检索。

## 3. MEMORY.md 的特殊处理：截断而非省略

**【事实】** `MEMORY.md` 是常驻注入的，但它可能长大。OpenClaw 的处理是：

```text
若 MEMORY.md 超过 bootstrap 文件预算:
  - 磁盘上的文件保持完整(intact)
  - 但注入进 model context 的那份副本被截断(truncated)
```

注意「磁盘完整、注入截断」这个区分很重要——它意味着**被截掉的部分仍然能被 `memory_search` 检索到**，只是不再常驻。这是一种优雅降级：体积失控不会丢数据，只会把超出部分从「常驻」降级为「按需」。

文档给出的应对建议：
- 把细节挪回 `memory/*.md`，`MEMORY.md` 只留耐久摘要
- 或抬高 bootstrap 上限

可观测手段：
- `/context list`、`/context detail`：看 raw vs injected 体积差
- `openclaw doctor`：体检

## 4. 相关预算配置

**【事实】** 涉及的配置键（默认值已按官方 `concepts/context` 文档校正，2026-06-07）：

| 配置键 | 含义 | 默认值 |
|---|---|---|
| `agents.defaults.bootstrapMaxChars` | 单个 bootstrap 文件注入上限 | `20000` |
| `agents.defaults.bootstrapTotalMaxChars` | 全部 bootstrap 文件合计上限 | `60000` |
| `agents.defaults.bootstrapPromptTruncationWarning` | 注入被截断时是否提示（`off`/`once`/`always`）| `always` |

> **校正说明**：本专题初版曾记到 total 上限有 `60000` / `150000` 两种说法（后者来自第三方 masterclass）。官方 `concepts/context` 文档确认默认是 **`60000`**，`150000` 系第三方资料口径误差，已弃用。关键不变：**bootstrap 有硬上限，记忆文件体积直接和每轮 token 成本挂钩。**

## 5. 不同 runtime / session 的差异

### 5.1 native Codex runtime 的注入方式不同

**【事实】** 在 native Codex runtime 上，`MEMORY.md` 不会被直接粘进每一轮：

```text
当 memory 工具可用时, Codex 的每轮会在 turn-scoped 的协作 developer
instructions 里收到一条简短的 workspace-memory note,
引导它按需调用 memory_search / memory_get。
```

也就是说，Codex 路线更激进地走「检索优先」——连 `MEMORY.md` 都不常驻，只给一句「需要就去查」的提示。

**【事实】** 一个例外：裸 `/new`、`/reset` 这种 turn，runtime 可以把最近的 daily memory 作为「一次性 startup-context 块」prepend 到那一轮，作为冷启动补偿。

### 5.2 sub-agent / minimal 模式裁剪

**【事实】** 记忆相关的 guidance（system prompt 里这一段名为 **Memory Recall**）在 **minimal prompt 模式（子 agent 用）里被省略**——minimal 模式明确砍掉 Memory Recall、OpenClaw Self-Update、Model Aliases、User Identity、Assistant Output Directives、Messaging、Silent Replies、Heartbeats 等。子 agent session 只注入 `AGENTS.md` 和 `TOOLS.md`，把人格（`SOUL.md`）和偏好（`USER.md`）这类文件按设计过滤掉。

> 设计含义：子 agent 是「干一件具体活」的临时工，不需要主人格和用户偏好，给它们反而是污染和浪费。这和 19 篇里「minimal mode 砍掉若干 section」是同一套裁剪逻辑——**按 agent 的职责裁剪它该记得的东西。**

### 5.3 主 session vs 群组/共享 session

**【事实】** `MEMORY.md` 在一处资料中被描述为「只在主、私有 session 加载，不进共享/群组上下文」，以保护机密。但官方 memory 概念页本身并未区分主/群 session。

> **【推断】** 这是隐私边界设计：私有长期记忆（可能含用户个人偏好、私密决定）不应泄露进多人群组会话。但具体在哪些 runtime 生效、粒度如何，需要源码确认。

## 6. 小结：这一刀切在哪

```text
常驻(贵, 每轮付费)         按需(便宜, 用时才付)
─────────────────         ───────────────────
MEMORY.md(策展长期)        所有历史 daily notes
今天 + 昨天 daily          被截断的 MEMORY.md 尾部
SOUL/USER/AGENTS/TOOLS     (通过 memory_search / memory_get)
```
