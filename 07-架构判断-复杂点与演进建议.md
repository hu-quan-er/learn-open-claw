# OpenClaw 架构判断、复杂点与演进建议

## 1. 总体判断

OpenClaw 的架构方向是对的，而且是少见地“把 agent 平台当平台来做”的项目。  
从公开仓库结构和 docs 看，它已经形成了比较清楚的层次：
- 接入层：gateway / channels / pair / UI
- 状态层：queue / sessions / groups / presence / memory
- 执行层：agents / jobs / tools / providers
- 扩展层：plugins / plugin-sdk / extensions / ACP
- 安全层：wizard / sandbox / identity / trusted pairing

这套结构的优点很明显，但复杂度也相应增长。

## 2. 明确的优点

### 2.1 控制面集中

gateway 被做成中心层，这是很正确的。  
它让协议、身份、server methods、pair、实时接入能收束在统一边界内。

### 2.2 状态模型成熟

session/group/queue/compaction/presence 这组抽象，说明团队已经在解决真正的 agent 平台问题，而不是停留在“把 LLM 接起来”。

### 2.3 扩展意识很强

`plugin-sdk` 和 `extensions` 的存在说明平台不是封闭的。  
这会极大影响后续生态能力。

### 2.4 安全不是补丁

pair、wizard、安全文档都单列，说明安全边界被结构化考虑了。

## 3. 主要复杂点

### 3.1 状态收敛难度高

只要一个平台同时支持：
- 多入口
- 多 session
- 多 group 分支
- 远端设备
- 插件
- agent 自主运行

那么状态一致性就是第一难题。  
queue 能缓解，但不能消灭所有复杂性。

### 3.2 抽象层次容易重叠

例如：
- `channels` 与 `channel plugins`
- `plugins` 与 `extensions`
- `agents` 与 `ACP agents`
- `memory` 与 `session compaction`

这些边界如果实现中不够克制，就会出现“概念上分开，代码上交叉”的问题。

### 3.3 安全边界跨层

安全不是单个模块能解决的：
- pair 属于接入层
- wizard 属于执行层
- sandbox 影响工具层
- identity 影响控制层
- queue 影响状态一致性

这意味着安全 bug 往往是跨层组合问题，而不是单函数 bug。

### 3.4 平台演进时的协议稳定性压力

既然 gateway 采用内部 RPC adapters 和 TypeBox schemas，那么一旦 UI、插件、远端设备都依赖这些 schema，协议演化就会越来越昂贵。

## 4. 最可能出现的维护风险

### 4.1 Gateway 变成超大总线

如果所有新能力都先挂到 gateway，长期看可能会把它做成“系统中什么都知道”的巨型中心。

### 4.2 Session 与 Group 的边界漂移

早期定义得再好，随着 feature 增长，也可能出现：
- 某些逻辑写在 session，其实应该写在 group
- 某些回退逻辑写在 group，其实应该属于 session snapshot

### 4.3 插件上下文过宽

插件越强，平台越开放；但插件一旦拿到过宽上下文，边界就会被侵蚀。

### 4.4 安全策略在不同入口不一致

只要入口够多，最容易出现的就是：
- CLI 可以，UI 不可以
- 配对设备可以，渠道插件不可以
- ACP agent 可以，embedded agent 不可以

这类“不一致”往往比纯缺权限更难查。

## 5. 我认为最值得继续深挖的源码问题

### 5.1 Queue 命令类型是否足够收敛

要看是否所有 mutation 真都走 queue，还是存在旁路写状态的情况。

### 5.2 Compaction 与 Memory 是否职责清晰

如果两者都在做“保留上下文”，职责很容易互相侵蚀。

### 5.3 Gateway schema 与内部对象模型是否过紧耦合

如果协议对象几乎等于内部状态对象，后续演进会很痛苦。

### 5.4 Plugin SDK 是否足够窄

好的 SDK 应该暴露能力，不暴露内部实现细节。

## 6. 如果我是维护者，我会重点守住什么

### 6.1 守住 queue 的唯一 mutation 通路

这是整个一致性地基。

### 6.2 守住 session/group 的抽象边界

不要让 feature 压力把它们重新糊成一团。

### 6.3 守住 plugin-sdk 的最小暴露面

一旦 SDK 过宽，未来收口会非常难。

### 6.4 守住 gateway 的协议纪律

新增 server methods 时必须要求：
- schema 明确
- 权限边界明确
- 是否改状态明确
- 是否需要进 queue 明确

## 7. 对二次开发者的建议

如果你要基于 OpenClaw 二开，不建议一上来就动核心 session 或 queue。  
优先切入点应该是：
- 新 provider plugin
- 新 channel plugin
- 新 extension
- 新 UI 面板
- 新 server method，但尽量不改底层状态模型

因为这些点对核心侵入较小，学习曲线也更可控。

## 8. 结论

OpenClaw 最有价值的地方，是它已经把 agent 平台的几个真正难点放到了一张架构图里：
- 状态
- 接入
- 执行
- 扩展
- 安全

最需要警惕的地方，也是这五块之间的耦合逐渐增加。  
如果后续演进能继续守住边界，它会是一套很强的平台内核；如果边界被 feature 压力冲垮，复杂度会迅速反噬。
