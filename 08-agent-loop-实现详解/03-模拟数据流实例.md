# 模拟数据流实例

下面用一个接近真实实现的例子，把 OpenClaw 的 `agent loop` 从入口到落盘完整走一遍。

场景：
- 用户通过 Telegram 给 agent 发消息
- 需求是“先排查，不改代码”
- 当前 session 没有并发 run

用户消息：

```text
帮我在 workspace 里定位 payment timeout 的根因，先别改代码。
```

## 1. 入站消息进入 Gateway

Gateway 先拿到一条入站消息，抽象上可以想成：

```json
{
  "channel": "telegram",
  "chatType": "direct",
  "userId": "tg_1832001",
  "text": "帮我在 workspace 里定位 payment timeout 的根因，先别改代码。"
}
```

这时系统先做的不是调模型，而是决定这条消息应该落到哪个 session。

## 2. 路由到 `sessionKey`

假设该用户命中了主会话：

```text
sessionKey = agent:main:main
sessionId  = sess_8d3a
runId      = run_52bf
lane       = session:agent:main:main
```

同时 `agent` RPC 很快就能给调用方回一个“已接受”：

```json
{
  "runId": "run_52bf",
  "acceptedAt": 1777747200000
}
```

注意这里不是最终结果，只是“这轮 run 已进入受控执行流程”。

## 3. 进入 `agentCommand`

`agentCommand` 会准备运行时参数，例如：
- 解析默认模型
- 解析 thinking/verbose/reasoning 等开关
- 加载当前可用 skills snapshot
- 准备把 run 交给 `runEmbeddedAgent`

可以把这一步理解成：

```text
“把一条用户消息，升级成一轮正式 agent run”
```

## 4. 进入 queue

然后 `runEmbeddedAgent` 把这轮 run 放进队列：

```text
session lane: session:agent:main:main
global lane : main
```

如果此时同一个 `sessionKey` 已经有别的 run 在跑：
- 这轮 run 会等待
- typing indicator 仍然可以先发
- 真正执行会在前一轮结束后开始

本例假设没有排队，直接开始。

## 5. 准备 workspace、prompt、tools

`runEmbeddedAttempt` 开始准备这轮执行：

```json
{
  "workspaceDir": "~/.openclaw/workspace",
  "sandbox": "off",
  "model": "anthropic/claude-sonnet-4-6",
  "timeoutMs": 120000
}
```

这一步会做几件关键事：
- 解析 workspace
- 构建 system prompt
- 读取 bootstrap/context files
- 加载 skills
- 构造 OpenClaw 自己的工具集
- 打开 `SessionManager`
- 拿 session transcript 写锁

工具集可能类似：

```text
read
exec
edit
write
message
browser
sessions_send
sessions_spawn
gateway
...
```

注意这些不是直接用 Pi 默认工具，而是 OpenClaw 自己注入和过滤后的工具集合。

## 6. 打开持久化状态

此时磁盘上大致有两层状态。

第一层，`sessions.json`：

```json
{
  "agent:main:main": {
    "sessionId": "sess_8d3a",
    "updatedAt": 1777747195000,
    "chatType": "direct",
    "providerOverride": null,
    "modelOverride": null
  }
}
```

第二层，`sess_8d3a.jsonl`：

```json
{"type":"session","id":"sess_8d3a","cwd":"~/.openclaw/workspace","timestamp":1777747100000}
{"type":"message","id":"m1","role":"user","content":"之前的一轮用户消息 ..."}
{"type":"message","id":"m2","role":"assistant","content":"之前的一轮回答 ..."}
```

OpenClaw 不是从零开始对话，而是在现有 transcript 上续写。

## 7. 建立 `AgentSession`（agent runtime）

接着会调用 agent runtime：

```text
createAgentSession(...)
```

然后 OpenClaw 立刻订阅这个 session 的事件：

```text
subscribeEmbeddedAgentSession(...)
```

这一步之后，Pi 内部的事件已经能被 OpenClaw 监听并转发。

## 8. 正式 prompt

随后真正把这轮用户输入送进去：

```text
session.prompt("帮我在 workspace 里定位 payment timeout 的根因，先别改代码。")
```

从这里开始，内层 agent loop 由所选 agent runtime 驱动（默认内置 openclaw harness）。

## 9. 第 1 轮内层推理

模型先产出一段助手说明，然后决定调用工具：

```text
assistant delta:
  我先搜索相关代码和超时关键字。

tool call:
  exec("rg -n \"payment timeout|ETIMEDOUT|timeout\" .")
```

OpenClaw 对外会发出类似流事件：

```json
{"runId":"run_52bf","stream":"lifecycle","data":{"phase":"start"}}
{"runId":"run_52bf","stream":"assistant","data":{"delta":"我先搜索相关代码和超时关键字。"}}
{"runId":"run_52bf","stream":"tool","data":{"phase":"start","tool":"exec"}}
```

## 10. 工具返回

假设 `exec` 返回：

```text
src/payments/retry.ts:88: timeoutMs: 3000
src/payments/client.ts:41: request timeout
src/payments/queue.ts:112: Payment timeout after retry budget exhausted
```

OpenClaw 会继续发：

```json
{"runId":"run_52bf","stream":"tool","data":{"phase":"end","tool":"exec"}}
```

同时这个 `toolResult` 会进入 transcript，不只是暂存在内存里。

## 11. 第 2 轮内层推理

模型看到 tool result 后，继续决定读文件：

```text
tool call:
  read("src/payments/retry.ts")
```

拿到文件后，它可能得出：
- 单次 timeout 只有 3 秒
- retry/backoff 预算和上游 SLA 不匹配
- queue 层把超时错误统一包装了，导致表象像“随机 timeout”

## 12. 最终 assistant reply

当模型认为已经足够回答时，会输出最终回复：

```text
根因大概率在 `src/payments/retry.ts` 的超时配置与重试预算不一致。
当前超时阈值偏小，而 queue 层又会把多次失败统一折叠成 payment timeout。
按你的要求，我先不改代码。建议下一步核对上游接口平均响应时间与 retry/backoff 配置。
```

此时 OpenClaw 会继续做一层整形：
- 过滤不该给用户看的静默 token
- 如果 messaging tool 已经替系统发过同样内容，则去重
- 组装最终 payloads

## 13. `agent.wait` 看到的结果

如果另一个调用方在等这轮 run，最终可能拿到：

```json
{
  "status": "ok",
  "runId": "run_52bf",
  "startedAt": 1777747200100,
  "endedAt": 1777747209400
}
```

## 14. transcript 里的实际新增内容

这轮 run 结束后，`sess_8d3a.jsonl` 里会新增近似这样的条目：

```json
{"type":"message","id":"m3","role":"user","content":"帮我在 workspace 里定位 payment timeout 的根因，先别改代码。"}
{"type":"message","id":"m4","role":"assistant","content":[{"type":"tool_call","name":"exec","args":"rg -n ..."}],"parentId":"m3"}
{"type":"message","id":"m5","role":"toolResult","content":"src/payments/retry.ts:88: timeoutMs: 3000 ...","parentId":"m4"}
{"type":"message","id":"m6","role":"assistant","content":[{"type":"tool_call","name":"read","args":"src/payments/retry.ts"}],"parentId":"m5"}
{"type":"message","id":"m7","role":"toolResult","content":"file contents ...","parentId":"m6"}
{"type":"message","id":"m8","role":"assistant","content":"根因大概率在 ...","parentId":"m7"}
```

这里的关键不是字段细节，而是：
- tool call 和 `toolResult` 会被持久化
- transcript 是树，不是简单数组
- 未来下一轮 prompt 会基于这份状态重建上下文

## 15. 如果中途触发 compaction

假设这时上下文已经非常长，OpenClaw/Pi 可能插入 compaction 路径：

```text
context overflow
  -> compaction_start
  -> 把旧内容摘要成 compaction entry
  -> 保留最近 tail
  -> 重新 retry 当前 run
  -> compaction_end
```

这里有两个关键保护：
- tool call 与对应 `toolResult` 不会被错误拆开
- retry 前会重置内存里的部分 buffer，避免重复输出

## 16. 这个例子真正说明了什么

这个例子里，OpenClaw 的核心价值不在“它也会调用 `rg` 和 `read`”，而在于：
- 先把消息安全地归到正确的 session
- 用 lane + write lock 保证这轮 run 不撞车
- 用所选 agent runtime 执行单轮 agent 内环
- 把 tool/assistant/lifecycle 结构化回流
- 把结果可靠写回 transcript

也就是说：

```text
一次 OpenClaw agent loop = 一次“可排队、可观测、可恢复、可持久化”的受控运行。
```
