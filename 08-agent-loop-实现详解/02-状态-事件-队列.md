# 状态、事件与队列

## 1. 真正的稳定边界不是一次模型调用

OpenClaw 这一层最重要的稳定边界有三类：
- `sessionKey/sessionId`
- lane queue
- transcript 持久化

如果只盯“这轮模型说了什么”，会错过整个系统真正防乱序、防重复、防上下文损坏的机制。

## 2. `sessionKey` 和 `sessionId` 分工

官方 session deep dive 明确区分了两层：

1. `sessionKey`
   - 表示“当前落在哪个对话桶里”
   - 负责路由与隔离
   - 例如 `agent:<agentId>:<mainKey>`、群聊 key、cron key

2. `sessionId`
   - 表示“这个桶当前指向哪份 transcript”
   - reset、idle 过期、每日切分后会切到新的 `sessionId`

所以：

```text
sessionKey = 路由身份
sessionId  = 当前正在续写的 transcript 文件身份
```

## 3. 两层持久化

官方文档把 session 持久化拆成两层：

1. `sessions.json`
   - `sessionKey -> SessionEntry`
   - 存当前 `sessionId`、更新时间、开关、token 计数、override 等

2. `<sessionId>.jsonl`
   - append-only transcript
   - 有树结构，entry 带 `id + parentId`
   - 存真实对话、tool call、tool result、compaction summary

这个结构很重要，因为它说明 OpenClaw 没把“会话状态”简化成一段内存里的 message array。

## 4. 队列是第一层并发控制

官方 Queue 文档明确写出：
- 每个 session 都会进自己的 lane：`session:<key>`
- 默认还有一个全局 lane：`main`
- session lane 保证一个 session 同时只有一个活跃 run
- global lane 控制总体并发

也就是：

```text
先按 sessionKey 串行
再按全局并发上限限流
```

这能防住的问题包括：
- 同一 session 同时发两轮 agent run
- tool 与 transcript 写入互相穿插
- 多个入站消息近距离撞车

## 5. 写锁是第二层并发控制

官方 Agent Loop 文档还强调了一个常被忽略的点：
- transcript 写入还有单独的 session write lock
- 这个锁是 process-aware、file-based
- 作用是兜住那些绕开内存队列或来自其他进程的写入

因此 OpenClaw 不是只靠 JS 内存队列做一致性。

可以把它理解成：

```text
lane queue 负责“谁先跑”
file lock 负责“谁能写”
```

## 6. 事件流怎么分

OpenClaw 当前主要暴露三类 stream：
- `lifecycle`
- `assistant`
- `tool`

语义分别是：
- `lifecycle`: run 开始、结束、报错
- `assistant`: 模型文本增量、partial reply、最终 reply
- `tool`: 工具开始、更新、中间结果、结束

这个划分说明 OpenClaw 不把所有流式输出混成一坨 token，而是保留了“谁在说话、当前在干什么”的结构语义。

## 7. `subscribeEmbeddedPiSession` 的角色

Pi 自己有 session event；OpenClaw 自己有 delivery/event bus。

`subscribeEmbeddedPiSession` 就是两者之间的翻译层：
- 订阅 Pi 事件
- 转换成 OpenClaw 的 stream event
- 控制 block chunking
- 处理 reasoning stream
- 过滤静默输出
- 协助去重 messaging tool 的确认文本

所以你看到的“assistant delta”、“tool start/end”、“lifecycle end”，并不是直接从模型 SDK 裸透出，而是翻译后的平台事件。

## 8. 回复整形不是附属逻辑

OpenClaw 在 delivery 侧还做了几件很关键的整形工作：
- assistant 文本与 reasoning 合成最终 payload
- verbose 时可内联 tool summary
- 过滤 `NO_REPLY` / `no_reply`
- 去掉 messaging tool 已经发出的重复确认
- 如果工具失败但没有可见回复，可兜底生成错误提示

这说明 OpenClaw 的“最终用户看到什么”，并不等于模型最后吐出的原始文本。

## 9. Compaction 与 retry 也在 loop 内

官方文档明确给出 auto-compaction 的两个触发条件：

1. overflow recovery
   - 模型返回 context overflow 一类错误
   - 先 compact，再 retry

2. threshold maintenance
   - turn 成功结束后，`contextTokens > contextWindow - reserveTokens`
   - 触发压缩维护

另外还有两个实现细节很重要：
- compaction 会把旧内容压成持久化 `compaction` entry
- assistant tool call 与对应 `toolResult` 会被成对保留，不会被错误拆开

这意味着 compaction 不是“丢掉老消息”，而是“重写 transcript 的可见结构”。

## 10. `group` 的位置

当前公开实现细节里，运行时热路径更明显地围绕：
- `sessionKey`
- `sessionId`
- lane
- transcript

而不是围绕 `groupId` 这种对象直接展开。

更稳妥的理解是：
- `group` 依然是上层多分支、多上下文的业务抽象
- 但今天这条 embedded agent runtime 主链路，工程重心更落在 session 路由和 transcript 维护上

## 11. 这层最该记的结论

```text
OpenClaw 的 agent loop 之所以稳定，不是因为 LLM 很聪明，
而是因为 session、lane、write lock、transcript、compaction 这些状态机制先被做好了。
```
