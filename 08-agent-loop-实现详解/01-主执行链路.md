# Agent Loop 主执行链路

## 1. 先抓住“谁负责什么”

OpenClaw 的 `agent loop` 不是单个 `while (true)`。

更准确的分层是：

```text
OpenClaw 外层：
  入口解析、session 路由、队列、写锁、工具策略、streaming、delivery、持久化

agent runtime 内层（可插拔）：
  一次 turn 的模型推理、tool call、tool result 注入、继续推理、直到结束
```

所以 OpenClaw 并不是自己从零实现 agent 内环，而是把一个 `AgentSession` 嵌入进自己的平台状态机里。这个内层 runtime 现在是**可插拔**的：

- **`openclaw`**（内置默认）：在 OpenClaw 进程内驱动模型循环，transcript 即 canonical 真相。
- **`codex`**：把底层 session 交给 Codex app-server，**Codex 自管 thread、tool continuation 与 compaction**，OpenClaw 退化为「镜像 transcript + 桥接事件 + 提供 dynamic tools」。`openai/*` 模型在 `auto` 模式下会自动解析到 codex。
- 还有 `claude-cli`、`copilot`、`acp` 等。通过 `agentRuntime.id`（model/provider 作用域）选择。

下面描述的热路径以内置 `openclaw` runtime 为主；runtime 归属不同会改变「谁拥有 thread 与 compaction」，但不改变 OpenClaw 外层的 session/queue/delivery 骨架。

## 2. 官方文档明确给出的主链路

官方 `Agent Loop` 文档把高层流程写得很清楚：

1. `agent` RPC 校验参数，解析 `sessionKey/sessionId`，持久化 session 元数据，然后立刻返回 `{ runId, acceptedAt }`
2. `agentCommand` 继续执行真正的 agent run
3. `runEmbeddedAgent` 负责排队、建 session、订阅事件、超时控制、返回 payloads
4. `subscribeEmbeddedAgentSession` 把 runtime 里的事件桥接成 OpenClaw 自己的 `agent` stream
5. `agent.wait` 通过 `waitForAgentRun` 等待 `runId` 结束

可以把它记成：

```text
接入层入口
  -> 运行时编排
  -> agent session 建立
  -> runtime 内层推理循环
  -> 事件桥接
  -> 回复整形与持久化
```

## 3. 真实热路径

按公开 runtime/harness 文档（`concepts/agent-runtimes`、`plugins/codex-harness`）与源码路径，最核心的调用顺序可以还原为：

```text
agent / CLI
  -> agentCommand
  -> runEmbeddedAgent()                       src/agents/pi-embedded-runner/run.ts
  -> runEmbeddedAttempt()                       src/agents/pi-embedded-runner/run/attempt.ts
  -> createAgentSession()                       agent runtime
  -> subscribeEmbeddedAgentSession()               src/agents/pi-embedded-subscribe.ts
  -> session.prompt(effectivePrompt, { images })
  -> agent runtime 处理完整 agent loop
  -> OpenClaw 收集 payloads + usage + stream
```

这里最关键的一步是：

```text
await session.prompt(...)
```

从这一刻开始，真正的“模型发消息、调工具、吃 toolResult、继续生成”的内环由 agent runtime 驱动。

## 4. `agentCommand` 做什么

`agentCommand` 不是低级执行器，而是 OpenClaw 的运行时入口包装层。

它至少负责：
- 解析模型、thinking、verbose、trace 等默认参数
- 加载 skills snapshot
- 调用 `runEmbeddedAgent`
- 在 embedded loop 没有正常发出结束事件时，补发 `lifecycle end/error`

这说明 `agentCommand` 的职责是“让一次 run 进入受控生命周期”，而不是直接碰 transcript 细节。

## 5. `runEmbeddedAgent` 做什么

这是 OpenClaw 外层编排的中心。

它负责：
- 对同一个 session 的 run 做串行化
- 把 run 再放进全局 lane，限制总体并发
- 解析 provider/model/auth profile
- 创建或打开 agent session
- 订阅 runtime 事件并转成 OpenClaw 的流
- 安装超时与 abort
- 生成最终 payloads 和 usage metadata

它不是一次普通函数调用，更像“一次受控 run 的容器”。

## 6. `runEmbeddedAttempt` 做什么

`runEmbeddedAttempt` 是一次具体尝试的准备层。

按公开文档，它会做这些事：
- 解析 workspace
- 解析 sandbox 信息
- 加载 resource loader / settings manager
- 载入 skills 并注入环境与 prompt
- 处理 bootstrap/context files
- 构造 system prompt
- 准备工具集
- 打开 `SessionManager`
- 最后调用 `createAgentSession(...)`

所以真正“把一次 agent turn 准备到可运行”的工作，主要不在 `agentCommand`，而在 `runEmbeddedAttempt`。

## 7. 工具不是直接裸传给 runtime

这是 OpenClaw 实现里很关键的一点。

公开 runtime/harness 文档明确写到（针对内置 openclaw runtime）：
- OpenClaw 自己构造工具管线
- `splitSdkTools()` 会把 `builtInTools` 置空
- 所有工具通过 `customTools` 注入

也就是：

```text
builtInTools = []
customTools = OpenClaw 适配后的工具集合
```

这么做的意义是：
- 统一做 sandbox 约束
- 统一做 tool allow/deny policy
- 注入 OpenClaw 自己的 messaging/browser/session/cron/gateway 等工具
- 针对不同 provider 做 schema 兼容和异常处理

## 8. 事件桥接是第二个核心

`subscribeEmbeddedAgentSession()` 不是小辅助函数，而是 OpenClaw 与 agent runtime 之间的桥。

它把 Pi 的事件翻译成 OpenClaw 的流：
- tool 相关事件 -> `stream: "tool"`
- assistant 增量 -> `stream: "assistant"`
- 生命周期 -> `stream: "lifecycle"`

文档列出的事件类型包括：
- `message_start` / `message_update` / `message_end`
- `tool_execution_start` / `tool_execution_update` / `tool_execution_end`
- `turn_start` / `turn_end`
- `agent_start` / `agent_end`
- `compaction_start` / `compaction_end`

所以 OpenClaw 的“实时输出能力”本质上不是模型 SDK 自带的 UI，而是这个桥接层在重发事件。

## 9. 主链路时序图

```text
用户消息 / RPC
  -> agent
  -> agentCommand
  -> runEmbeddedAgent
  -> session lane 排队
  -> global lane 排队
  -> runEmbeddedAttempt
  -> createAgentSession
  -> subscribeEmbeddedAgentSession
  -> session.prompt(...)
  -> runtime 内层循环
       -> assistant delta
       -> tool call
       -> tool result
       -> assistant final
  -> OpenClaw 整形 payloads
  -> transcript / sessions store 更新
  -> lifecycle end
```

## 10. 最后怎么记

如果只记一句话：

```text
OpenClaw 的 agent loop = “用 queue + session + event bridge 包装 agent-runtime AgentSession 的运行”。
```
