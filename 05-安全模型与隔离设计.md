# OpenClaw 安全模型与隔离设计

## 1. 为什么安全在这个项目里是一级主题

OpenClaw 不是纯本地单用户脚本，而是一个具备：
- 远端配对
- 多入口接入
- 工具调用
- agent 自主执行
- 可能的外部运行时

的平台。  
这类系统如果安全边界模糊，风险会比普通聊天产品高很多。

从公开 docs 看，OpenClaw 至少把以下安全面显式文档化了：
- Pair
- Wizard security
- Sandboxing / agent workspace

这已经说明团队不是把安全留到最后补，而是把它写进系统结构里。

## 2. Pair：可信设备接入边界

官方 Pair 文档表明：
- 系统通过 pairing token 建立可信接入
- 支持二维码等方式把设备或端接入同一系统
- 配对不是普通登录页，而是控制面安全的一部分

这意味着 OpenClaw 在“多端接入”问题上不是简单共享一个 API key，而是显式建立设备信任关系。

从架构上说，Pair 解决的是：
- 哪个设备能接入你的 OpenClaw
- 这个设备以什么身份接入
- 这个连接如何被控制面识别与撤销

## 3. Wizard：受限环境而不是普通 agent

`Wizard security` 文档给出的信息非常关键：
- wizard 运行在 detatched 或受控的环境中
- 有更严格的 cwd 和 allowlist 限制
- 拥有单独的 prompt / 边界
- 默认不应获得普通 shell 级别的自由能力

这说明 OpenClaw 并没有假设“所有 agent 都值得信任”。  
相反，它把某些 agent 或工作流放进更狭窄的执行边界。

这是成熟平台才会做的事。

## 4. Sandboxing 的架构意义

虽然公开资料里不一定逐文件展开所有实现，但从安全和 workspace 文档可以推断出几个明显原则：
- agent 可执行能力必须受限
- 可访问目录必须受限
- 不同运行模式的权限面不同
- 平台控制面需要知道当前能力边界

对于 agent 系统，sandbox 不只是“防止删库”，还包括：
- 防止跨 workspace 越权读取
- 防止插件拿到过大能力
- 防止远端接入执行本地高权限操作
- 防止 assistant identity 与真实权限边界脱节

## 5. Identity 与权限的关系

`assistant-identity.ts` 放在 gateway 邻域，很容易让人注意到：  
身份不是单纯 UI 展示，而是安全边界的一部分。

因为在 agent 平台里，身份会影响：
- 用户如何理解这个 agent 的能力
- 这个 agent 被允许做什么
- 这个 agent 的提示词和工具边界是什么
- UI 与远端设备怎样显示权限风险

分析推断：
- OpenClaw 很可能把“身份、能力、入口”绑定在控制面，而不是让每个 agent 自己定义一套

## 6. Queue 也是安全组件

很多人只把 queue 当并发控制，但对 OpenClaw 来说它同时也是安全稳定组件。

因为串行化的状态变更能显著降低：
- 并发权限绕过
- 多端同时修改造成的脏状态
- 工具输出与用户输入交错造成的错误决策

如果安全边界依赖当前 session/group 状态，那么 queue 的一致性直接影响安全性。

## 7. 多端接入下的主要风险

OpenClaw 这类平台主要风险集中在：

### 7.1 入口扩张风险

入口一多，就会出现：
- 哪些入口能发高权限命令
- 哪些入口只能订阅不能执行
- 哪些入口能发起配对或创建 session

### 7.2 插件越权风险

插件和扩展如果拿到过宽平台上下文，就可能绕过设计边界。

### 7.3 Agent 自主执行风险

agent 如果能自行调用工具、继续运行 job、跨 session 访问资源，就必须有严格边界。

### 7.4 远端设备风险

pairing 一旦被滥用，远端设备等于获得本地控制面的入口。

## 8. OpenClaw 的安全设计优点

从公开资料看，至少能确认以下优点：
- 配对被单独设计，不是顺手拼上去的
- wizard 被隔离，而不是默认给满权限
- gateway 中心化，有利于统一施加控制策略
- session/group/queue 的中心化状态模型让审计与收敛更可行

## 9. 仍需重点核查的源码点

如果后续拿到完整本地源码并继续深挖，优先核查：
- Pair token 的生成、存储、失效和撤销逻辑
- wizard 的 cwd / allowlist / tool whitelist 实现
- plugin-sdk 暴露给插件的上下文是否过宽
- gateway server methods 是否存在入口权限不一致
- session/group 是否能跨边界读取不该读的数据

## 10. 结论

OpenClaw 的安全模型不是“给每个函数加权限判断”，而是更偏结构性安全：
- 入口边界由 gateway 和 pair 控制
- 执行边界由 wizard / sandbox 控制
- 状态一致性由 queue / session 模型控制
- 扩展边界由 plugin-sdk 和 extensions 体系控制

这说明它已经进入“平台安全”范畴，而不是普通应用安全范畴。
