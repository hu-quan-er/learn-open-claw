# OpenClaw 仓库结构与模块地图

## 1. Workspace 级观察

根据官方 `pnpm-workspace.yaml`，当前 workspace 至少纳入以下范围：
- 根包 `.`
- `ui`
- `packages/*`
- `extensions/*`

这个配置很关键，因为它直接说明仓库不是单包项目，而是多组件协同：
- 根包负责主程序或平台整合
- `ui` 负责控制面界面
- `packages/*` 负责可复用 SDK 或子能力
- `extensions/*` 负责扩展生态

## 2. 顶层目录的角色分工

从官方仓库首页能直接看到的顶层目录包括：
- `apps`
- `docs`
- `extensions`
- `packages`
- `src`
- `ui`

可以做出相对稳妥的模块判断：

| 目录 | 直接角色 | 分析判断 |
| --- | --- | --- |
| `src` | 平台核心源码 | 主控制面、网关、会话、agent 运行时大概率都在这里 |
| `ui` | 控制面前端 | docs 已明确 Control UI 是 Vite + Lit |
| `packages` | SDK / 可复用包 | 提供跨模块复用，而不是把所有代码堆进 `src` |
| `extensions` | 插件生态 | 平台扩展面落地目录 |
| `apps` | 外部应用或宿主 | 可能承载 demo、客户端或部署载体 |
| `docs` | 设计文档 | 是理解源码的最佳入口之一 |

## 3. 根入口层

根 `package.json` 能直接确认两点：
- 包名是 `openclaw`
- 暴露的命令入口是 `openclaw.mjs`

这表明 CLI 或主进程启动并不是从 `src/index.ts` 这类典型 Node 入口直接暴露，而是先通过一个顶层 `mjs` 包装入口进入系统。

分析推断：
- `openclaw.mjs` 很可能承担启动参数分发、环境准备、动态导入或 bootstrap 的职责
- 把入口单独放在根目录，有利于发布 CLI 时保持 Node 侧稳定入口，不必暴露内部 ts 路径结构

## 4. `src` 目录的结构含义

从 `src` tree 页面可直接看到多个一级目录。公开可见的核心块包括：
- `agents`
- `channels`
- `cli`
- `gateway`
- `memory`
- `node-host`
- `plugins`
- `plugin-sdk`
- `services`
- `utils`
- `wizard`

这几个名字已经足以说明平台是按职责边界切层，而不是按“页面/功能点”堆叠。

### 4.1 `agents`

这是 agent loop、agent 会话控制、agent 指令派发的核心区。  
结合 `docs/pi.md`，这里至少会和以下概念相关：
- agent session
- embedded pi agent
- job lifecycle
- command dispatch
- session subscription

### 4.2 `channels`

`channels` 的存在说明 OpenClaw 明确把“消息入口/出口”做成一级模块，而不是把 Slack/Discord/CLI/remote 等入口散落在业务代码里。

### 4.3 `gateway`

这是控制面中枢。根据 `docs/gateway-architecture.md`：
- 当前采用 RPC adapters
- gateway 统一协议边界
- 使用 TypeBox schema 保持类型安全

从 `src/gateway` 目录还能直接看到更细分的结构：
- `protocol/`
- `server/`
- `server-methods/`
- `voiceclaw-realtime/`
- `agent-prompt.ts`
- `assistant-identity.ts`
- `boot.ts`
- `boot-pair.ts`
- `boot-voiceclaw.ts`

这已经足够说明 gateway 不只是 HTTP handler，而是一个同时承担：
- 协议定义
- 服务启动
- 方法注册
- 语音实时接入
- 不同启动模式分支

的中心层。

### 4.4 `memory`

单独抽出 `memory`，说明系统有独立的长期/半长期上下文策略，而不是把所有上下文都塞在 session transcript 里。

### 4.5 `plugins` 与 `plugin-sdk`

把“插件实现”和“插件 SDK”拆开，是平台成熟度的重要信号。  
这意味着：
- 平台作者自己实现的插件放在 `plugins`
- 给第三方或扩展作者使用的开发接口放在 `plugin-sdk`

### 4.6 `wizard`

安全文档里专门有 `Wizard security`，说明 `wizard` 不是普通功能目录，而是受限运行环境的一部分。

## 5. `ui` 目录的含义

官方 Control UI 文档明确指出：
- Control UI 是基于 `Vite`
- UI 技术栈使用 `Lit`

这说明 OpenClaw 没有选择 React/Next 这一套传统 web app 堆栈，而是更偏向轻量、组件化、与平台控制面更贴近的前端实现。

分析推断：
- 这通常意味着 UI 不是内容型产品前端，而是偏运维/控制台/工作台性质
- 与 gateway 的 RPC 模型天然更契合

## 6. `packages` 与 `extensions` 的平台信号

单看目录命名就能确认两件事：

### 6.1 `packages` 说明代码在走“平台内模块复用”

如果一个项目只有 `src` 而没有 `packages`，通常说明它是单体应用。  
OpenClaw 有 `packages`，说明它已经把一部分稳定能力抽象成共享包。

### 6.2 `extensions` 说明扩展机制不是事后补丁

如果扩展体系只存在于 `src/plugins`，那往往意味着插件只是内部概念。  
OpenClaw 额外保留了 `extensions` 顶层目录，意味着扩展生态是仓库结构层面的第一公民。

## 7. 按“平台分层”重新看模块

更适合阅读 OpenClaw 的方式，不是按目录一层层点开，而是按平台层次理解：

```text
入口层
  openclaw.mjs / CLI / pairing / UI entry

接入层
  gateway / channels / RPC adapters / realtime

状态层
  queue / sessions / groups / presence / memory

执行层
  agents / tools / provider invocation / jobs

扩展层
  plugins / plugin-sdk / extensions / ACP / channels

控制与安全层
  wizard / sandbox / pair / identity / permission boundaries
```

这样读，比只看路径更不容易迷失。

## 8. 模块地图结论

OpenClaw 的目录结构传达出非常明确的工程观：
- 平台先于单功能
- 状态与接入先于 UI
- 扩展是一级能力，不是补丁
- 安全边界单独成层

这也是为什么它的源码更像“平台内核”而不是“应用业务层”。
