# 构建入口与数据来源

OpenClaw 的 system prompt 构建发生在 agent run 准备阶段，而不是模型调用之后的补丁逻辑。它是一次 run 开始前，把当前 agent 能力、workspace、工具、channel、sandbox、skills、memory 和 provider 特性合成到模型可见系统上下文里的过程。

## 1. 在 agent loop 中的位置

官方 Agent Loop 文档把一次 run 描述成：

```text
intake -> context assembly -> model inference -> tool execution -> streaming replies -> persistence
```

system prompt 属于 `context assembly` 阶段。也就是说，模型开始推理前，OpenClaw 已经完成了：
- session / workspace 解析
- skills snapshot 载入
- bootstrap/context files 收集
- tool catalog 与 tool policy 过滤
- model / provider / auth profile 解析
- sandbox 和 channel 能力解析

然后这些信息进入 `buildAgentSystemPrompt(...)`，渲染成当前 run 的 system prompt。

## 2. 核心入口

公开源码中的核心函数是：

```text
src/agents/system-prompt.ts
  buildAgentSystemPrompt(params)
```

从参数形状可以看出，它不是一个简单的字符串模板函数，而是一个运行时聚合器。重要输入包括：

| 输入 | 作用 |
| --- | --- |
| `workspaceDir` | 当前工作目录，用于 Workspace section 和路径约束 |
| `toolNames` / `toolSummaries` | 当前可见工具集合与摘要，用于 Tooling section |
| `contextFiles` | 注入的 workspace/bootstrap 文件 |
| `skillsPrompt` | skills 子系统生成的可用技能元信息 |
| `runtimeInfo` | agentId、host、OS、repoRoot、model、shell、channel、capabilities 等 |
| `sandboxInfo` | 是否沙箱、容器路径、workspace access、elevated exec 等 |
| `docsPath` / `sourcePath` | OpenClaw docs/source 的本地或远程位置 |
| `extraSystemPrompt` | 运行时追加的 group/subagent 上下文 |
| `promptMode` | `full`、`minimal`、`none` 三种 prompt 体积模式 |
| `promptContribution` | provider 贡献的 section override、stable prefix、dynamic suffix |

分析推断：OpenClaw 把 system prompt 构建函数做成显式参数输入，是为了让 prompt 结果可测试、可缓存、可诊断；它没有让各模块随意拼字符串，而是把拼装集中在一个边界内。

## 3. 上游数据来源

system prompt 的信息来源大致分成六类。

### 3.1 Agent / session 运行态

这部分来自当前 run 的执行上下文：
- 当前 agent id
- 当前 session
- 当前 channel
- 当前 model/default model
- thinking / reasoning 级别
- active background exec sessions

这些信息用于 `Runtime`、`Reasoning`、`Messaging`、`Assistant Output Directives` 等 section。

### 3.2 Workspace 与 bootstrap 文件

OpenClaw workspace 是 agent 的默认工作区，也是被注入 project context 的主要来源。默认涉及：
- `AGENTS.md`
- `SOUL.md`
- `TOOLS.md`
- `IDENTITY.md`
- `USER.md`
- `HEARTBEAT.md`
- `BOOTSTRAP.md`
- `MEMORY.md`

源码里还会对文件做排序和分流：
- 稳定文件进入 stable project context
- `HEARTBEAT.md` 这类高波动文件进入 dynamic project context
- `BOOTSTRAP.md` 有专门的 bootstrap pending 补充逻辑

这说明 workspace 文件不是普通附件，而是 system prompt 的一部分。

### 3.3 工具集合与工具策略

Tooling section 的工具列表不是静态写死的。源码里先定义 core tool summary，再用当前 `toolNames` 做过滤和排序。

核心点：
- 模型只看见当前 policy 允许暴露的工具
- 工具名大小写要精确匹配
- 额外工具会按稳定顺序加入
- `TOOLS.md` 只是使用建议，不控制工具是否可用

这把“模型知道什么工具”和“模型能不能调用什么工具”分开了：prompt 负责说明，tool policy 才负责硬约束。

### 3.4 Skills 与 memory

Skills 不会把所有 `SKILL.md` 全量塞进 system prompt。OpenClaw 注入的是 compact skills list，包含 name、description、location，然后要求模型在确实适用时再读对应的 `SKILL.md`。

Memory section 由 `buildMemoryPromptSection(...)` 条件构建，是否出现取决于 prompt mode、可用工具和配置。

这两个设计都在控制 system prompt 体积：让模型知道“可访问什么”，但不默认加载所有细节。

### 3.5 Channel / messaging 能力

OpenClaw 不是纯 CLI coding agent，它可以从 Telegram、Signal、Discord、webchat 等 channel 收消息和发消息。因此 system prompt 会按 channel capability 注入：
- 当前回复是否自动路由到 source channel
- 是否必须用 `message(action=send)` 交付可见输出
- inline buttons 是否可用
- reaction guidance 是否可用
- Control UI/webchat 是否支持 `[embed ...]`
- cross-session 和 sub-agent 消息如何发送

这部分让模型理解“回复文本”和“真正交付到用户所在渠道”之间的差别。

### 3.6 Provider 贡献

Provider 不能随便替换整份 OpenClaw prompt，但可以通过 `ProviderSystemPromptContribution` 做三类有限贡献：
- 替换少数核心 section：`interaction_style`、`tool_call_style`、`execution_bias`
- 在 cache boundary 上方加入稳定前缀
- 在 cache boundary 下方加入动态后缀

这是一种很克制的扩展点：允许 provider/model-family 调优，但不让 provider 接管 OpenClaw 的运行时协议。

## 4. 输出形态

源码中的构建结果不是一次性 `join` 所有 section。它先构造 stable prefix，再写入内部 cache boundary，然后追加 dynamic suffix。

大致结构：

```text
stablePrefix:
  identity
  Tooling
  Tool Call Style
  Execution Bias
  provider stable prefix
  Safety
  OpenClaw Control / Self-Update
  Skills / Memory
  Workspace / Documentation / Sandbox
  stable Project Context
  Silent Replies
  CACHE BOUNDARY

dynamicSuffix:
  dynamic Project Context
  Control UI / Messaging / Voice
  Group Chat Context or Subagent Context
  Reactions
  provider dynamic suffix
  Heartbeats
  Runtime / Reasoning
```

这就是 OpenClaw system prompt 最核心的工程结构：不是“写一段提示词”，而是“渲染一份当前 run 的执行配置快照”。
