# 运行时条件注入

OpenClaw system prompt 的核心不是“有哪些固定文字”，而是“哪些 section 在什么条件下出现”。这使它能服务多 agent、多 channel、多 provider、多 sandbox 策略，而不需要为每种场景维护一整套完全不同的 prompt。

## 1. promptMode 分级

官方文档和源码都能看到三种 `promptMode`：

| mode | 用途 | 保留内容 |
| --- | --- | --- |
| `full` | 主 agent 默认模式 | 几乎全部 section |
| `minimal` | sub-agent 常用模式 | 工具、安全、workspace、sandbox、时间、runtime、注入上下文等核心内容 |
| `none` | 极简模式 | 基础身份句，可附带 model identity |

`minimal` 会省略大量主会话专用内容，例如 Skills、Memory Recall、OpenClaw Self-Update、Model Aliases、User Identity、Reply Tags、Messaging、Silent Replies、Heartbeats。

设计意图很明确：子 agent 不需要完整人格、记忆、渠道协议和自维护能力；它需要的是足够完成隔离任务的工具、workspace 和边界信息。

## 2. 工具集合条件化

`Tooling` section 依据当前可用工具生成，而不是固定列全量工具。

源码里的流程可以概括为：

```text
toolNames
  -> trim / normalize / dedupe
  -> 按 core toolOrder 输出已知工具
  -> 额外工具排序后追加
  -> 用 coreToolSummaries 或 toolSummaries 填说明
  -> 生成 Tooling 文本
```

由此得到几个行为：
- 配置或 policy 禁掉的工具不会出现在工具说明中
- 未知扩展工具仍可出现，但顺序稳定
- core 工具有更精确的内置摘要
- 工具名 casing 会尽量保留调用方传入的形式

分析推断：这套设计同时服务“行为正确”和“cache 稳定”。工具顺序如果每次乱变，即使语义相同，也会破坏 prompt prefix cache。

## 3. Skills 按需加载

有可用 skills 时才出现 `Skills` section；没有 eligible skills 时整段省略。

Skills section 不直接包含技能正文，而是包含 skills prompt，也就是技能索引。模型被要求：
- 只在明显适用时读取
- 选择最具体的技能
- 一开始最多读一个
- 不猜技能路径
- 外部 API 写入要批量、避免 tight loops、尊重 429/Retry-After

这个设计有三个好处：
- system prompt 小
- 技能读取可追踪
- 具体技能的高优先级指令只在需要时进入上下文

## 4. Memory 条件化

Memory section 通过 `buildMemoryPromptSection(...)` 生成。它会被 `promptMode=minimal` 和 `includeMemorySection=false` 过滤。

这说明 memory 不是 system prompt 的必备常量，而是主 agent 运行时能力。子 agent 一般不应该天然继承完整记忆，否则容易跨任务污染和增加 token 成本。

## 5. Workspace 文件注入

官方 System Prompt 文档说明，OpenClaw 会把 bootstrap files trimmed 后放到 Project Context。官方 Context 文档还给出 `/context list`、`/context detail`，用于观察每个注入文件和 tool schema 的 token 成本。

源码层面有三个关键机制。

### 5.1 固定顺序

源码定义了 context file order：

```text
AGENTS.md
SOUL.md
IDENTITY.md
USER.md
TOOLS.md
BOOTSTRAP.md
MEMORY.md
```

如果还有其他文件，则按 basename/path 排序。这个顺序不是为了阅读美观，而是为了让相同 workspace 的 prompt prefix 尽量 byte-identical。

### 5.2 动态文件分流

`HEARTBEAT.md` 被识别为 dynamic context file。稳定 context files 放到 stable prefix，动态 context files 放到 cache boundary 后的 `Dynamic Project Context`。

这避免 heartbeat 内容频繁变动时把前面的工具、skills、workspace、docs 等稳定内容一起打爆 cache。

### 5.3 内容清洗

源码对某些 heartbeat 默认 prompt 文本做了清洗，避免 provider 或模式对特定 prompt-policy 片段敏感。同时会压缩过多空行。

分析推断：这说明 system prompt builder 不只是拼接文件内容，它还承担 provider 兼容和 prompt hygiene 的责任。

## 6. Bootstrap 条件化

`bootstrapMode` 可能是：
- `none`
- `limited`
- `full`

当 full bootstrap pending 时，prompt 会告诉模型：
- `BOOTSTRAP.md` 已在 Project Context 中，或需要读取它
- 先处理 bootstrap，再正常回复
- 不要用 generic greeting
- 不能完成时说明 blocker，不要假装完成

当 limited 时，它会说明当前 run 不能安全完成完整 bootstrap，只能做可安全推进的部分，并给出下一步。

这把“新 workspace 的初始化仪式”纳入了系统协议。

## 7. Sandbox 条件化

只有 `sandboxInfo.enabled` 为真时才出现 `Sandbox` section。它会按运行时状态拼出：
- 工具是否在 Docker 中执行
- container workspace dir
- host mount source
- workspace access
- browser bridge
- host browser 是否允许
- elevated exec 是否可用
- `/elevated full` 是否可用
- full access 不可用时的原因

特别重要的是路径指导：

```text
file tools -> host workspace path
exec commands -> sandbox container path
prefer relative paths
```

如果没有这层说明，模型很容易把 host path 传给 container shell，或把 container path 传给 file tool。

## 8. Channel / messaging 条件化

Messaging section 受多个条件影响：
- `message` 工具是否可用
- `sessions_spawn` / `sessions_yield` / `subagents` 是否可用
- `sourceReplyDeliveryMode` 是否是 `message_tool_only`
- 当前 channel 是否支持 inline buttons
- 当前 channel 是否是 Slack、Discord、webchat 等
- `messageToolHints` 是否存在

几个典型分支：
- `message_tool_only`：可见输出必须通过 `message(action=send)`，最终普通文本保持私有
- 普通 source reply：最终回复自动路由到 source channel
- inline buttons 可用：提示 buttons 参数格式
- inline buttons 不可用但 channel 有能力配置：提示配置项
- 有 `sessions_yield`：等待子 agent 完成时用 yield，不轮询

这说明 OpenClaw 的 prompt 会根据交付媒介改变模型行为，而不是强迫所有 channel 使用同一套输出格式。

## 9. ACP / sub-agent 条件化

源码里有 ACP harness 相关条件：
- `acpEnabled`
- sandbox 是否启用
- `sessions_spawn` 是否可用
- runtime capabilities 是否包含 thread-bound ACP spawn
- channel 是否为 Discord

当满足条件时，prompt 会把“在 claude code/cursor/gemini/opencode 中做”等请求识别为 ACP harness intent，并指导使用 `sessions_spawn(runtime:"acp")`，而不是本地 PTY exec 或普通 subagent 工具。

如果 sandboxed runtime 中 ACP harness 被阻断，prompt 会明确说明 ACP harness spawns blocked，应使用 `runtime:"subagent"`。

这是一个典型的运行时能力塑形：同一句用户请求，在不同 runtime capability 下应该走不同工具路径。

## 10. Output directives 条件化

`Assistant Output Directives` 受 `sourceReplyDeliveryMode` 影响：
- 如果是 message-tool-only，则附件、voice note、native quote/reply 都走 message tool 字段
- 否则使用 `MEDIA:`、`[[audio_as_voice]]`、`[[reply_to_current]]` 等文本指令

`Control UI Embed` 只在 webchat/control UI 场景出现，并且明确禁止把 `[embed ...]` 用在非 web channel 或 local filesystem path。

`Voice (TTS)` 只有存在 TTS hint 时出现。

`Reactions` 只有 channel 开启 reaction guidance 时出现，并按 minimal/extensive 两种模式给不同策略。

## 11. Silent reply 条件化

Silent Replies 在 full prompt 中常见，但当 `sourceReplyDeliveryMode=message_tool_only` 或 `silentReplyPromptMode=none` 时会被抑制。

原因很实际：如果可见输出已经通过 message tool 交付，普通最终文本可能只是私有控制面返回；继续强行要求 `NO_REPLY` 反而会干扰 channel-specific flow。

## 12. Runtime 末尾注入

Runtime section 总是在最终拼装阶段靠后追加，包含：
- agent
- host
- repo
- os/arch
- node
- model/default_model
- shell
- channel/capabilities
- thinking
- active process sessions
- reasoning 状态

它放在 cache boundary 后方，是因为这些值更容易随 run、channel、model override 或后台进程状态变化。

## 13. 条件注入的设计价值

OpenClaw 的条件注入让 system prompt 同时具备：
- 小：没有能力就不写对应 section
- 准：当前 run 有什么能力就写什么能力
- 稳：稳定 section 尽量保持固定顺序和固定文本
- 可扩展：新 channel、新工具、新 provider 可以贡献局部 guidance
- 可治理：高风险能力如 gateway/self-update/elevated exec 必须显式出现在 prompt 和 tool policy 中

这也是它能支持多入口、多 agent、多工具、多 provider 的关键。
