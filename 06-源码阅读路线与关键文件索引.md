# OpenClaw 源码阅读路线与关键文件索引

## 1. 阅读原则

OpenClaw 不适合从某个随机 `index.ts` 开始硬读。  
更有效的方法是按“入口 -> 控制面 -> 状态层 -> 运行时 -> 扩展层”的顺序推进。

## 2. 第一轮：先建立地图

第一轮阅读只做三件事：
- 看 workspace 结构
- 看顶层目录角色
- 看官方 docs 的总架构

建议顺序：
1. `package.json`
2. `pnpm-workspace.yaml`
3. 仓库首页 tree
4. `docs/architecture.md`
5. `docs/gateway-architecture.md`

目标不是理解实现，而是先知道系统大块头在哪里。

## 3. 第二轮：把入口和控制面读通

建议顺序：
1. `openclaw.mjs`
2. `src/gateway/boot.ts`
3. `src/gateway/boot-pair.ts`
4. `src/gateway/boot-voiceclaw.ts`
5. `src/gateway/protocol/*`
6. `src/gateway/server/*`
7. `src/gateway/server-methods/*`
8. `src/gateway/assistant-identity.ts`
9. `src/gateway/agent-prompt.ts`

这一轮要回答的问题：
- 主程序是怎么起来的
- 启动模式怎么分支
- gateway 怎样定义协议
- 哪些 server methods 是主要入口
- identity/prompt 在控制面中的位置是什么

## 4. 第三轮：把状态层读通

建议顺序：
1. `docs/queue.md`
2. `docs/sessions-and-groups.md`
3. `docs/session-compaction.md`
4. 与 queue 有关的核心实现
5. 与 session/group 相关的状态对象或服务
6. `src/memory/*`

这一轮要回答的问题：
- 所有 mutation 是否真的都经过 queue
- group 与 session 的数据边界怎样划
- compaction 在什么时机触发
- memory 与 session transcript 怎样协作

## 5. 第四轮：把 agent runtime 读通

建议顺序：
1. `docs/pi.md`
2. `src/agents/*`
3. 与 job lifecycle 相关的实现
4. 与 tool/provider 调用相关的连接代码

重点关注这些关键词：
- `createAgentSession`
- `runEmbeddedPiAgent`
- `waitForAgentJob`
- `subscribeEmbeddedPiSession`
- `agentCommand`

这一轮要回答的问题：
- agent 与 session 的绑定关系是什么
- job 生命周期如何建模
- 工具输出如何回流状态层
- agent 能否跨 session / group 工作

## 6. 第五轮：把扩展体系读通

建议顺序：
1. `docs/plugins/plugin-sdk-overview.md`
2. `docs/plugins/provider-plugins.md`
3. `docs/plugins/channel-plugins.md`
4. `docs/acp-agents.md`
5. `src/plugin-sdk/*`
6. `src/plugins/*`
7. `src/channels/*`
8. `extensions/*`

这一轮要回答的问题：
- 插件注册点在哪里
- provider 和 channel 的抽象是否对称
- ACP agent 如何接入主平台
- 扩展代码能拿到多少平台上下文

## 7. 第六轮：补安全与端侧

建议顺序：
1. `docs/pair.md`
2. `docs/wizard-security.md`
3. `docs/sandboxing.md` 或相关安全文档
4. `docs/control-ui.md`
5. 与 `wizard`、`pair`、`ui` 相关实现

这一轮要回答的问题：
- 远端配对怎样建立和撤销信任
- wizard 有哪些硬性限制
- UI 依赖 gateway 的哪些能力
- 权限模型如何向用户暴露

## 8. 一份最小可执行的阅读清单

如果只给自己半天时间，建议读以下文件和文档：

```text
package.json
pnpm-workspace.yaml
openclaw.mjs
docs/architecture.md
docs/gateway-architecture.md
docs/pi.md
docs/queue.md
docs/sessions-and-groups.md
docs/session-compaction.md
docs/control-ui.md
docs/pair.md
docs/wizard-security.md
```

读完这组，已经足够建立全局心智模型。

## 9. 最值得做的源码笔记方式

建议不要边读边记散点笔记，而是维护四张表：

### 9.1 入口表

记录：
- 启动入口
- 启动模式
- 对应 boot 文件
- 对应 gateway 方法注册点

### 9.2 状态表

记录：
- session 相关对象
- group 相关对象
- queue 命令类型
- presence 结构
- compaction 触发点

### 9.3 扩展表

记录：
- provider plugin 注册接口
- channel plugin 注册接口
- plugin-sdk 对外暴露接口
- extensions 中已有样例

### 9.4 安全表

记录：
- pair token 生命周期
- wizard 限制项
- sandbox 边界
- 高风险 server methods

## 10. 读源码时最容易踩的坑

### 10.1 先盯模型调用

这会让你忽视真正复杂的地方：控制面和状态层。

### 10.2 把 session 当成唯一核心对象

实际上 group 更可能是上层业务语义。

### 10.3 只看 `src`

OpenClaw 这种项目必须配合 `docs` 一起读，否则成本极高。

### 10.4 忽略 UI 与 gateway 的关系

Control UI 不是边缘功能，它反过来能帮助你理解平台暴露了哪些核心能力。

## 11. 最终建议

用三次阅读法最稳：
1. 第一次只建地图，不抠细节
2. 第二次只追主链路，不展开全部分支
3. 第三次才围绕一个问题做深挖，例如 queue、一种 plugin，或 pair 安全

对 OpenClaw 这种平台源码，先求结构正确，再求细节完整。
